# ckit_cell_grids.csv

Per-cell grid geometry extracted from the CKIT-RF classified rasters, so a Google Earth Engine
export can pin its output grid to each cell's exact georeferencing with no post-hoc resampling.
Generated by `scripts/extract_ckit_grids.py`.

## Source

The classified interpretation rasters under `data/raw/rf_class_maps/`, named
`rf_class_reviewer_<name>_grid_<id>_sample_<n>_sensor_<sensor>_target_<year>_opt_<y1>_<y2>.tif`.
254 rasters were found. 253 are geometry-valid (10 m, north-up), covering 180 unique cells.

## Columns

`cell_id, crs, x_scale, x_shear, x_trans, y_shear, y_scale, y_trans, width, height, file`, one row
per unique cell.

- `cell_id` is the `grid_<id>` value as an integer, zero-padded to a width-5 string
  (`str(int(id)).zfill(5)`), so it joins against the GEE grid asset id column and the `cell_id`
  column of `interpreted_cells_by_bracket.csv`, both width-5 strings. An unpadded or integer id
  produces a silent zero-match join in GEE, so the padding is retained.
- `x_scale, x_shear, x_trans, y_shear, y_scale, y_trans` are the affine transform in GEE
  `crsTransform` order `[xScale, xShear, xTranslate, yShear, yScale, yTranslate]`, which is
  rasterio's `Affine (a, b, c, d, e, f)`. Feed these six values, together with `crs`, `width`, and
  `height`, to the per-cell export so the prediction grid matches the reference grid exactly.

## Validation applied

- Every written row is EPSG:5070, north-up, and 10 m (`x_scale == 10`, `y_scale == -10`,
  `x_shear == 0`, `y_shear == 0`), asserted per raster.
- Label encoding was checked against the crosswalk (0->4, 1->6, 2->7, 3->3, 4->5, 5->8, 20->1,
  30->2, 50->10, 62->9) plus the two valid-but-excluded values 10 (unknown abstention) and 13
  (other_no_change). No raster carried a value outside that set.
- The 72 double-interpreted cells (two reviewers, one `cell_id`) were checked for grid agreement
  across reviewers. All 72 match, so keeping the first occurrence is safe.

The script refuses to write the CSV if any in-scope raster has an unmapped label value or any
double-interpreted cell has mismatched grids, so this file cannot ship with an unresolved encoding
or alignment problem.

## Grid sizes vary

Among the 180 unique cells the grid dimensions are not uniform: 156 are 337x337, 13 are 337x338,
and 11 are 337x336. The width is always 337, and the height varies by a pixel. A single fixed
337x337 export would misalign the 24 cells that are not 337x337, so each cell's `width` and
`height` must be pinned from its own row.

## Excluded records

One raster was excluded from the CSV for bad geometry:
`rf_class_reviewer_Robert_grid_17800_sample_16_sensor_Landsat_target_2004_opt_2003_2005_epsg5070.tif`.
It is 30 m and 113x113 (Landsat, not Sentinel-2), and it is the only record on the out-of-scope
2003_2005 bracket, so it is out of scope for a 10 m pin. Its `grid_17800` id appears nowhere else,
so excluding it drops no in-scope cell. This leaves 180 cells, matching
`interpreted_cells_by_bracket.csv`.

## Note on ids beyond the study count

As documented for `interpreted_cells_by_bracket.csv`, some cell ids exceed the 21,561-cell study
count (the id space indexes a larger tiling). Grid membership cannot be confirmed from the rasters
alone, so verify the ids against the GEE grid asset, where a non-member id will simply fail to join.
